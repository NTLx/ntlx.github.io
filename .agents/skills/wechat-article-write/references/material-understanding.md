# 材料理解增强

本文件定义 Step 1.8：把 `materials.md` 和 `blog-memory.md` 变成可直接喂给 `ljg-writes` 的 `understanding-brief.md`。目标是提高文章质量，不追求最快完成。

## 适用范围

| 策略 | 处理 |
|---|---|
| `reader-response` | 必须执行 Step 1.8 |
| `news-digest` | 默认不执行完整 Step 1.8；只在用户要求深度复盘时按本文件裁剪使用 |
| `tutorial` | 默认不执行；只在原文概念过密时调用 `ljg-plain` 改善解释 |

## 输入

- `posts/{date-slug}/materials.md`
- `posts/{date-slug}/blog-memory.md`（若已生成）
- 用户的写作意图或限制

## 强制调用

`reader-response` 必须调用以下 ljg 技能：

| 技能 | 用途 | 写入 brief 的内容 |
|---|---|---|
| `ljg-qa` | 抽出核心问题链 | `## 核心问题链`，包含问题、结论、边界 |
| `ljg-think` | 对中心论点纵向下钻 | `## 中心论点下钻`，给出文章真正要打的一句话 |

## 条件强制

根据材料类型额外调用：

| 条件 | 技能 | 用途 |
|---|---|---|
| 主材料是长文、英文、复杂文本 | `ljg-read` | 结构地图、关键问题、跨领域连接 |
| 文章是领域、趋势、产业、工具链分析 | `ljg-rank` | 找背后的生成器，供正文和 SLOT 00 信息图使用 |
| 文章解释行业、角色、产品行为，或材料里有方案争论、约束错配、旧解释被当硬事实 | `ljg-constraint` | 找硬/软/自设约束、解空间、身份约束和可松动的旧解释 |
| 技术或术语密度高 | `ljg-plain` | 把关键解释压到聪明读者能跟上的版本 |
| 有 1-2 个必须解释的核心概念 | `ljg-learn` | 概念来源、反对对象、直觉画面 |
| 输入是论文、arXiv、研究 PDF | `ljg-paper` | 论文命题、方法、边界、可用启发 |
| 文章要讲论文脉络 | `ljg-paper-river` | 问题演化史和新旧解法位移 |
| 输入是书或书摘 | `ljg-book` | 核心问题、旧共识、作者位移 |
| 观点争议大或需要反方测试 | `ljg-roundtable` | 反方、弱点、可承认的边界 |
| 项目、公司、商业模式分析 | `ljg-invest` | 秩序创造机器判断 |
| 文章围绕英文词或命名 | `ljg-word` | 词源意象和核心语义 |

未触发的条件技能不要调用；不要为了凑流程制造分析。

## 输出格式

保存为 `posts/{date-slug}/understanding-brief.md`：

```markdown
# Understanding Brief

## 原始材料结构

## 核心问题链

## 中心论点下钻

## 关键概念白话化

## 领域生成器

## 约束与解空间

## 反方与边界

## 可写成正文的判断

## 可视觉化的节点

## 写作契约
```

## 写作契约

`## 写作契约` 是 Step 2 的输入合同，必须用短句写清：

- 这篇文章只打一件什么事；
- 必须回答哪些问题；
- 必须承认哪些边界；
- 哪些背景资料要自然织入正文；
- 哪些站内旧文适合联动；
- 哪些约束、解空间或约束错配需要成为正文判断；
- 哪些节点应该变成 `SLOT_IMG`；
- 明确禁止写成摘要、翻译、资讯搬运或教程。

Step 2 调用 `ljg-writes` 时，必须把写作契约作为最高优先级内容输入。

## 失败处理

- 强制技能不可用时，先向用户说明缺失技能和影响，不要假装已完成理解增强。
- 条件技能不可用时，在 `understanding-brief.md` 对应小节写明未执行原因。
- 如果 Step 1.8 发现材料超过单篇承载，回到 `pipeline-overview.md` 的多文章拆分规则，先征求用户确认。

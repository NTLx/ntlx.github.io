前一版文章有个标题，我越看越觉得不对。

《让 agent 去洗澡》。

这个说法很顺口，但它把事情说反了。Jason Liu 原文里的场景不是“Agent 去洗澡”，而是他自己去洗澡，Codex 留在电脑前继续处理 Amazon 退款。人离开，Agent 没停。等他回来，退款办完了。

这个差别不是抠字眼。

“让 Agent 去洗澡”听起来像一个段子，重点落在荒诞感上，好像我们在讲 AI 多聪明、多拟人、多会替人生活。但原文真正有意思的地方恰恰相反：Agent 没有去生活，它只是被放进了一个能继续运行的工作回路里。它有任务，有等待策略，有可触达的界面，有判断条件，也有一个明确的完成标准。

我重新读 Jason 的《Codex-maxxing》，发现前一版文章还有一个更大的偏差：我把 memory 写得太重了。

memory 当然重要。Jason 用 Obsidian vault 做共享记忆，让 Agent 把人、项目、决定、待办、关闭的 loop 写进可审查的 Markdown 文件里，再用 Git diff 看它记住了什么。这非常漂亮。但原文的中心不是“memory 是唯一基础设施”。原文更准确的中心词是另一个：operating loop。

Jason 写得很清楚：真正改变他行为的，是学会给工作一个 operating loop：持久线程、共享记忆、能操作电脑的工具、过程中可以 steering 和 resume 的方式，以及一个能审查产物本身的界面。

这句话比“九条 Codex 技巧”重要得多。

因为它说的不是功能清单，而是工作形态的变化。

## 标题错在哪里

![](https://cdn.jsdelivr.net/gh/NTLx/Pic@master/wechat-articles/2026-05-24-codex-maxxing-img-01-comparison-subject-correction.png)

中文里“让 X 去洗澡”的默认意思，是 X 执行“洗澡”这个动作。所以“让 Agent 去洗澡”会让读者本能地困惑：Agent 为什么要洗澡？这和 Codex 有什么关系？

原文的动作主体其实很清楚：

Jason 的包裹被偷了。Amazon 告诉他，想找人工客服大概要等 25 分钟。他创建了一个带 `@computer` 的 thread，告诉 Codex：

- 每 5 分钟检查客服有没有进来
- 如果客服进来了，就尽力帮我争取退款
- 一旦客服回复，把检查频率改成每 1 分钟，方便更快响应

然后他去洗澡。

等他出来，退款完成。

这个故事的重点不是“AI 替我聊客服”这么简单。它真正让人一震的地方是：这类工作本来就不是一直需要人的高强度思考，它更像一段被等待切碎的流程。你要等客服上线，判断对方有没有回复，给出下一句回应，再继续等。人在里面最痛苦的部分，不是写那几句话，而是被迫守在那里。

Codex 之所以能接住这件事，不是因为它突然变成了万能秘书，而是因为 Jason 给了它一个能跑起来的小回路：

- 完成标准：退款办完
- 执行节律：先每 5 分钟检查，客服出现后每 1 分钟检查
- 操作通道：通过 `@computer` 接触 Amazon 客服界面
- 过程判断：客服是否上线，是否已经回复
- 结果状态：退款是否完成

如果标题要表达这件事，应该是“我去洗澡，让 Agent 继续干活”，或者更直接一点：“我离开之后，Agent 还能继续推进工作吗？”

这不是修辞问题，是理解问题。标题一旦把动作主体说反，整篇文章就容易滑向“AI 很神奇”的故事；而 Jason 原文其实在讲“工作可以怎样被结构化成一个 Agent 能持续推进的 loop”。

## 原文讲的不是九个功能，而是一套回路

![](https://cdn.jsdelivr.net/gh/NTLx/Pic@master/wechat-articles/2026-05-24-codex-maxxing-img-02-comparison-chat-vs-loop.png)

很多人读这种文章，容易把它读成 tips 清单：

- pinned thread 很有用
- voice input 很方便
- steering 可以边做边改
- memory 要写到 Obsidian
- heartbeat 可以定时检查 Slack
- goals 要有测试
- side panel 能看产物

这些都对，但如果只停在这里，就还是在“功能评测”的层面。

Jason 真正关心的是：工作能不能在 prompt 之间继续活着。

我们平时用 AI，最熟悉的是聊天模式。你发一句，它回一句；你再发一句，它再回一句。球永远在两边来回传。只要球回到你手里，工作就停了。你去开会，停；你去吃饭，停；你去睡觉，停。第二天回来，你还得重新解释上下文，或者忍受一个已经变得臃肿、含混、快要压缩坏掉的对话。

Jason 想要的不是更聪明的聊天框，而是让工作有一个可以容身的地方。

这个地方由几层东西组成。

durable thread 负责连续性。一个重要 workstream 不再是一次性聊天，而是一个可以 pinned、可以 compact、可以反复回来的长期线程。它保留旧决定、偏好、历史背景，减少每次从零开始的损耗。

voice input 负责把粗糙想法喂进去。人很多真实意图其实很难打字，比如“Slack 里好像有个叫 Ben 的人提过这事，我记不清了，你去找找”。这句话用键盘输入会显得潦草，用语音说出来却很自然。Agent 需要的往往不是 polished prompt，而是足够多的真实上下文。

steering 负责中途修正。Agent 正在改网页时，你不必等它完全做完再说“这里小一点、文案不对、间距怪怪的”。你可以在它运行时继续补方向。这样任务不是“一问一答”，而是在执行过程中持续被塑形。

computer/browser use 和 connectors 负责接触真实世界。代码 repo 只是工作的一部分。Slack、Gmail、Calendar、浏览器、桌面 GUI，很多任务都发生在那里。Agent 如果只能在代码目录里打转，就接不住“等客服”“看评论”“查邮件”“上传文件”这些实际流程。

heartbeats 负责节律。线程不再只是等用户叫醒，而是可以每 15 分钟、每 30 分钟、每几个小时自己回来检查一次。它可以看 Slack 有没有新反馈，看 PR 有没有评论，看 Gmail 有没有需要处理的邮件。没有 heartbeat，线程是静止的；有了 heartbeat，线程才开始像一个会定期巡检的系统。

goals 负责终点和验证。弱目标是“把这个计划实现一下”。强目标是“迁移这个库，但必须通过原项目的测试套件”。测试、benchmark、复现用例、端到端流程，都是告诉 Agent 什么时候算做完的 oracle。没有验证的雄心，最后只会变成“看起来差不多”。

side panel 负责审查表面。Agent 生成的不是一段聊天回复，而是 Markdown、PDF、表格、slides、网页、Storybook、Remotion 动画。你要能看见它、标注它、评论它，Agent 也要能操作同一个对象。工作产物不应该离开对话之后再审查，审查本身就是 loop 的一部分。

这些东西合在一起，才是 operating loop。

所以重读这篇原文后，我觉得“九条技巧”这个说法反而有点遮蔽。Jason 不是在说“你看 Codex 又多了九个功能”。他是在说：当这些能力连在一起时，Agent 从一个回答者，变成了一个能持续推进任务的工作回路。

## Memory 重要，但不是唯一答案

![](https://cdn.jsdelivr.net/gh/NTLx/Pic@master/wechat-articles/2026-05-24-codex-maxxing-img-03-framework-memory-as-diff.png)

前一版文章把 shared memory 写成了主角。这个判断有道理，但过头了。

Jason 的 memory 做法确实值得认真看。他把一个 Obsidian vault 当作 Agent 的工作记忆：

```text
vault/
├── TODO.md
├── people/
├── projects/
├── agent/
└── notes/
```

顶层放一个 `AGENTS.md`，告诉 Agent：当你更理解某个人、推进了某个项目、关闭了某个 open loop，就更新对应页面。

这个设计厉害的地方不是“Obsidian”三个字，而是它把记忆从模型内部搬到了文件系统。

文件有几个好处。

第一，它可审查。Agent 说自己记住了什么并不重要，重要的是你能不能看到 diff。它写下“这个项目卡在某个 blocker 上”“这个人偏好短回复”“这个决定已经做出”，你都能在 Git 里看到。

第二，它可编辑。模型记忆常常像黑箱，你只能祈祷它记对了。Markdown 文件不一样，错了就改，乱了就删，必要时直接 revert。

第三，它可复用。一个 thread 学到的东西，不应该只困在那个 thread 的 transcript 里。下一个 thread、另一个 worker、三天后的你，都应该能读到同一个 notebook。

Jason 说他不想让 evergreen threads 在 conversation history 里悄悄积累 vibes。他希望它们把变化写下来：谁喜欢什么，项目等什么，哪个决定已经做了，哪个 loop 关掉了。

这句话很关键。memory 不是“AI 要有灵魂”，memory 是“工作要留下结构化痕迹”。

但也正因为如此，memory 不应该被写成其他能力的替代品。

没有 tools，Agent 只能记笔记，不能行动。没有 heartbeats，Agent 有记忆也不会自己定期醒来。没有 goals，它不知道什么时候算完成。没有 side panel，你很难审查产物本身。没有 steering，人也没法在执行中细调方向。

memory 是 loop 的沉淀层，不是整套 loop 本身。

这一点对我们自己的使用方式也很重要。很多人听到“给 Agent 建记忆”，第一反应是去找某个神奇插件，或者把一堆偏好塞进系统提示词。但 Jason 的启发更朴素：先让重要上下文有一个能被写入、被审查、被复用的位置。

它可以是 Obsidian vault，可以是 repo 里的 `docs/decisions/`，可以是项目根目录的 `AGENTS.md`，也可以是一组明确命名的 Markdown 文件。关键不是结构多漂亮，而是 Agent 知道什么该写、写到哪里、什么时候不要制造噪音。

## 从 prompt mindset 到 loop mindset

我读完这篇文章后，最大的提醒不是“我要多用 Codex 的某个功能”，而是我得换一个提问方式。

以前我写 prompt，默认是在请求一次回复：

“帮我看看这段代码。”

“把这篇文章总结一下。”

“给我写一个登录页面。”

这些请求不是不能用，但它们很容易把工作锁死在聊天框里。AI 给出一个结果，然后停下。接下来要不要继续，往哪继续，怎么判断好坏，全都回到我身上。

loop mindset 会先问另一组问题：

这件事能不能独立推进一小段？

它需要接触哪些系统？

它应该多久检查一次？

什么情况算完成？

中间遇到不确定性时，哪些可以自己判断，哪些必须回来问我？

产物写在哪里，我怎么审查？

哪些信息要沉淀到长期记忆里，哪些只是临时过程？

你会发现，这些问题一问出来，prompt 的形态就变了。

“帮我处理退款”太空。更像 Jason 的写法是：

“这个客服窗口大概要等 25 分钟。你每 5 分钟检查一次客服是否上线。上线后帮我争取退款。客服回复后改成每 1 分钟检查一次。退款完成、需要我提供敏感信息、或者对方拒绝退款时停下来告诉我。过程记录到这个文件里。”

这不是更长的 prompt 而已，而是把一件事拆成了一个可运行的回路。

同样，“帮我迁移这个库”也太空。更强的写法是：

“把这个 Python 库迁到 Rust。原库测试套件是验收标准。先建立最小可运行版本，再逐步补齐 API。每次修改后运行对应测试。测试通过数、失败原因、下一步计划写进 `migration-status.md`。如果遇到语义不确定的 API，列出来等我决定。”

这时 Agent 不再只是回你一段“迁移计划”，而是有机会持续推进。

当然，这不是说所有事情都该交给 Agent 自己跑。很多任务还不适合。涉及钱、账号、对外发送、不可逆操作、法律风险、隐私数据时，人必须保留最终确认权。Jason 的 Chief of Staff thread 也只是 draft reply，不直接发送。这个边界很重要。

但边界不是退回聊天框的理由。更好的方式是把权限写清楚：哪些可以做，哪些只能草拟，哪些必须停下来问。

## 这篇文章真正该叫什么

所以，如果要重写前一版文章，我不会再用“让 Agent 去洗澡”。

我会叫它：

**我去洗澡，让 Agent 继续干活。**

这个标题至少把动作主体讲对了。

但标题背后的重点还要再往前走一步：这不是洗澡故事，也不是退款故事，更不是“AI 终于能替我生活”的故事。

它讲的是一个很具体的工作方式变化：

以前我们等 AI 回复。

现在我们开始等 Agent 做完。

这两个“等”完全不一样。

等回复时，AI 是聊天对象。它说完一句，工作就停在你面前。

等做完时，Agent 是一个被放进回路里的执行者。它有上下文，有工具，有节律，有目标，有审查面。你可以离开，但工作不一定停。

Jason 原文最后那句话，我觉得比所有案例都重要：

Codex 得到越多可以记住、重访、检查和行动的地方，我的工作就越不会死在 prompt 之间。

这才是整篇文章的骨头。

不是 Agent 更像人了。

而是工作终于不必每次都死在聊天框里。

*你最近交给 AI 的任务里，有哪一件其实不是要一个回复，而是需要一个能继续运行的小回路？*
